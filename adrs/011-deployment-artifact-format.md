# Cosign-signed published tarball over SDK-on-host for Archetype A2 install

- Status: accepted; **default flip landed** (#249, 2026-07-06 — `install.sh` defaults to `--from-release`, `--from-source` gated behind `--i-accept-unverified-source`)
- Date: 2026-05-22
- Companion: `scripts/install.sh`, `.github/workflows/release.yml`, [README §Archetype A2](../README.md#archetype-a2-native-os-service-install-no-docker)
- Tracking issue: [#245](https://github.com/psmfd/agent-expertise-api/issues/245)
- Surfaced by: PR #241 (install/uninstall hardening track C) pre-design review by `dotnet-expert`, 2026-05-22; PR #248 shipped Option A as the pragmatic short-term default.

## Context and Problem Statement

Archetype A2 (native OS service install via `scripts/install.sh` / `install.ps1` — the alternative to `docker compose up` and Helm) runs `dotnet publish` on the install host. That requires the **.NET 10 SDK** (~700 MB), not just the ASP.NET Core runtime. The PR C brief framed dependency bootstrap as "runtime-only"; the pre-design review caught this as internally inconsistent.

There is also a **latent preflight bug**: `preflight()` checks for `Microsoft.AspNetCore.App 10.x` via `dotnet --list-runtimes`, but the script then invokes `dotnet publish`, which needs an SDK. An operator on a runtime-only host passes preflight and fails at `publish_app_staged`. PR #248 (PR C1) papered over this by installing the SDK on macOS via Homebrew under `--install-deps`. That is the right short-term move and the wrong long-term shape.

The decision: how does Archetype A2 obtain runnable bits?

## Considered Options

- **A. SDK-on-host + build-from-source-on-host.** Status quo after PR #248. `dotnet publish` runs on the operator's machine against the operator's git clone and the live NuGet supply chain. ~700 MB SDK; install-time RCE surface = the transitive NuGet + MSBuild graph (each package's `<package>.targets` is auto-imported MSBuild code with `Exec` authority); no signature verification anywhere on the install path. CVE-patch cadence for the runtime stays with the OS package manager (good).
- **B. CI pre-publishes a portable RID-agnostic tarball; `install.sh` cosign-verifies and extracts.** Publish happens in `release.yml` against a known-pinned SDK in an ephemeral runner with scoped `GITHUB_TOKEN`. CI emits a single portable tarball (`-p:UseAppHost=false`, no `--runtime`), a manifest JSON declaring the runtime floor sourced from the generated `runtimeconfig.json`, and a cosign `sign-blob` over the manifest. `install.sh` fetches both, verifies via `cosign verify-blob` against a script-constant OIDC identity, then `bsdtar`-extracts into `${STAGE_DIR}` and proceeds with the existing atomic swap + migrate-against-staged flow from ADRs/PRs #243/#244. Runtime CVE-patch cadence still owned by the OS package manager. The existing wrapper already falls through to `dotnet ExpertiseApi.dll` when no apphost is present (install.sh `write_wrapper`), so the launcher path is unchanged.
- **C. Self-contained deployment (`--self-contained true`).** ~80 MB on host, no host runtime required, but we ship the .NET runtime ourselves — every CVE in `Microsoft.AspNetCore.App` requires a new release + reinstall on every A2 host on our cadence, not the OS package manager's. Inverts the CVE-patch lifecycle in the wrong direction for an LTS service.

## Decision Outcome

**Chosen: Option B**, with Option A retained as an explicit `--from-source` opt-in (devs iterating on a clone; air-gapped operators).

Chosen because PR C1 (#248) has just shipped Option A as the pragmatic short-term default; the cosign keyless-OIDC ceremony already exists in this repo for the GHCR image, the Helm OCI artifact, and the openapi.json release asset (`release.yml` lines ~143–280; README §Supply-chain verification); and the two pending Linux bootstrap PRs (#246 Debian, #247 RHEL) are blocked behind a decision that determines whether their per-distro work targets "install an SDK from Microsoft package feeds (and pin the GPG fingerprint, and verify the .deb SHA, and branch on `$VERSION_ID`)" or "fetch and verify one signed tarball." Option B collapses the bootstrap surface from "platform-conditional SDK install" to "single signed-blob fetch" across all three OS branches and removes the install-time RCE surface (transitive NuGet + MSBuild `<package>.targets` with `Exec` authority — security-review estimates one to two orders of magnitude reduction).

Option C is rejected on CVE-lifecycle grounds and is not revisited absent an air-gapped or LTS-frozen-target requirement neither of which is currently in scope.

This ADR records the **commitment** to migrate to Option B. PR C1 currently runs Option A by definition (no signed tarball exists yet). The flip from `--from-source` to `--from-release` as the default is a single behavior change in `install.sh` and is tracked as a separate follow-up issue (filed alongside this ADR), gated on: (a) one release cycle of green CI publishing the signed tarball + manifest; (b) `scripts/verify-release.sh` shipping with a smoke test that exercises cosign-verify-then-extract end-to-end on macOS and Linux runners.

### Sub-decisions

**Distribution channel: GitHub Release asset (v1) → GHCR OCI artifact (v2).** Release assets are mutable by name — a `contents: write` adversary can swap `agent-expertise-api-v10.0.5.tar.gz` for an older legitimately-signed `v10.0.3.tar.gz`. GHCR OCI digests are content-addressed and eliminate that class structurally. Ship v1 on Release assets to reuse the existing `release.yml` `gh release upload` + cosign `sign-blob` ceremony verbatim; migrate to GHCR OCI artifacts as a follow-up once `oras` (or equivalent) is a paid-for install-time dependency. The downgrade-defense in the *signing scope* sub-decision below covers the v1 swap window.

**Signing scope: sign a manifest JSON, not the bare tarball.** A manifest `{schemaVersion, appVersion, gitSha, publishMode, rid, targetFramework, requiredRuntime: {name, minVersion, rollForward}, sdkUsedToPublish, buildTimestamp, artifacts: {tarballSha256, openapiSha256, modelManifest}}` covers: (a) version monotonicity — `install.sh` reads `appVersion` and refuses downgrade against `${VERSION_MARKER}` from PR #244 unless `--allow-downgrade` is passed; (b) cross-RID integrity — one signature spans all RID tarballs by content hash, so an adversary cannot mix-and-match across signed releases; (c) SBOM alignment — the publish-time SDK version, source git SHA, and NuGet lockfile hash are signed-attestable. `requiredRuntime.minVersion` is sourced from the generated `ExpertiseApi.runtimeconfig.json` (`runtimeOptions.framework.version`), not hand-set; otherwise it silently drifts.

**Verification UX: cosign required, no sha256 fallback.** `install.sh` under Option B's default path refuses to proceed without `cosign verify-blob` returning success. The `--from-source` opt-in is the documented escape hatch. Silent fallback from cosign-failure to sha256-only would defeat the trust-root migration. `cosign` itself becomes a host dep installed under `--install-deps` from a SHA-256-pinned sigstore/cosign release tarball (never `latest`); the OIDC identity (`https://github.com/psmfd/agent-expertise-api/.github/workflows/release.yml@refs/heads/main`) and issuer (`https://token.actions.githubusercontent.com`) are script constants — exact-match, no regexp (same rule the README already documents for openapi.json verification).

**`--from-source` retention: explicit flag, no auto-detect.** Auto-detecting "is this a git clone" via `[ -d .git ]` is fragile (worktrees, shallow extracts). Use an explicit `--from-source` flag; the SDK preflight (`command -v dotnet`) fires only on that path. The runtime preflight (`Microsoft.AspNetCore.App 10.x` via `dotnet --list-runtimes`) fires unconditionally — both paths run fdd. `--from-source` requires `--i-accept-unverified-source` to acknowledge the trust degradation (no NuGet/MSBuild signing on that path). The install-mode is recorded in a sibling of `${VERSION_MARKER}` so subsequent upgrades can detect a mode regression and warn.

**Independent GitHub Releases API cross-check (added during D3 implementation, #249).** Before downloading any release asset, `install.sh --from-release` queries `https://api.github.com/repos/{owner}/{repo}/releases/tags/v${version}` and asserts (a) `tag_name == v${version}`, (b) the five expected asset filenames (`*-portable.tar.gz`, `*.tar.gz.sha256`, `*.manifest.json`, `*.manifest.json.sig`, `*.manifest.json.pem`) are all present in the API's asset listing. Skippable via `--skip-release-api-crosscheck` for air-gap mirrors (deferred to #255). Rationale: the manifest's cosign signature covers cross-RID integrity and version monotonicity, but an asset-rename attack (uploading the legitimately-signed `v10.0.3` tarball under the `v10.0.5` filename) would still pass cosign verify-blob alone. Cross-checking the API binds the `${version}` requested on the CLI to the API's tag→assets mapping — a second trust path that an adversary with `contents: write` cannot forge without also compromising api.github.com. Cost: one HTTPS roundtrip per install. First-install policy: `--version vX.Y.Z` is REQUIRED on the first `--from-release` install; `--version latest` only permitted on upgrades. Forces the operator's trust commitment onto the CLI rather than being silently resolved by the API during a swap-asset-name attack window.

**Post-install marker layout (added during D3, #249).** Three sibling files in `${PREFIX}/`: (a) `.install-version` (existing, opaque `git describe` from PR #244); (b) `.install-mode` — `release\n` or `source\n` — primary input to upgrade-time mode-regression detection; (c) `.install-version-semver` — clean semver + manifest sha, release-mode only, primary input to downgrade defense; (d) `.install-history` — append-only audit trail with ts, mode, version, manifest sha, tarball sha, cosign identity, cosign version. All four written AFTER `atomic_swap` so a rollback path that runs `cleanup()` before `SUCCESS=1` never leaves markers that describe an install that did not commit.

## Consequences

- **Good** — eliminates the install-time SDK requirement on every A2 host; collapses #246/#247 Linux bootstrap from "platform-conditional Microsoft-feed SDK install" to "fetch + cosign-verify + bsdtar-extract" across all three OS branches.
- **Good** — closes the latent preflight bug (runtime-only host passes preflight then fails at publish): preflight's runtime assertion is now sufficient because `install.sh` never invokes `dotnet publish` on the default path.
- **Good** — reduces install-time RCE surface from O(transitive NuGet + MSBuild graph, dozens to low hundreds) to O(cosign + bsdtar + .NET runtime, single digits). One to two orders of magnitude.
- **Good** — closes the runtime-vs-publish-SDK version-skew class of install failure (manifest's `requiredRuntime.minVersion` is compared with `sort -V` against `dotnet --list-runtimes`, not the loose `10.x` regex).
- **Good** — inherits the existing in-repo cosign keyless-OIDC ceremony verbatim; no net-new trust-root or signing-key surface.
- **Bad** — `cosign` becomes a hard install-time dependency under `--install-deps`. Maintenance: a pinned `COSIGN_VERSION` + per-RID SHA-256 in `install.sh` constants requires manual quarterly review (dependabot does not bump shell-script constants).
- **Bad** — release pipeline becomes load-bearing for A2 upgrades. A release that publishes the image but fails to publish the signed tarball blocks Archetype A2 upgrades until republished. Mitigation: `--from-source` is the documented escape hatch.
- **Bad** — `--from-source` (Option A) is a second-class UX going forward: build-from-source remains supported but is not end-to-end signature-verified. The README must be explicit so it is not a surprise.
- **Bad** — air-gapped operators (Rekor/Fulcio unreachable) need an explicit `--allow-offline-verify` opt-in that consumes a pre-fetched Rekor bundle. Documented as a stated limitation; not silent degradation.
- **Bad** — portable single-tarball publish requires an explicit `<RuntimeIdentifiers>` list in the csproj for ONNX RID-native packing; without it the tarball crashes at startup on non-CI-runner platforms. Tracked as a precondition for the implementation PR (see Implementation notes).
- **Revisit if** — air-gapped or LTS-frozen-target requirement emerges (re-evaluate Option C scope); or the multi-RID matrix grows beyond what a single portable tarball can serve (would force `<RuntimeIdentifiers>` matrix expansion).

## Implementation notes (non-normative)

These details may drift; do not treat them as the ADR's normative content.

- **Portable RID-agnostic publish** is the recommended Option B shape: `dotnet publish src/ExpertiseApi/ExpertiseApi.csproj --configuration Release -p:UseAppHost=false -p:Deterministic=true -p:ContinuousIntegrationBuild=true -p:DebugType=embedded --output dist/publish` — no `--runtime`. Single tarball; launched via `dotnet ExpertiseApi.dll` through the existing wrapper fallback. ~5% startup penalty vs apphost (irrelevant for a long-lived service).
- **ONNX `<RuntimeIdentifiers>` precondition (HIGH).** Portable publish only ships RID-natives for runtimes declared in the csproj `<RuntimeIdentifiers>` property. Without an explicit list, `runtimes/<rid>/native/` will not contain non-CI-runner platforms and the tarball will crash at startup on other hosts. Add `<RuntimeIdentifiers>linux-x64;linux-arm64;osx-arm64;osx-x64;win-x64</RuntimeIdentifiers>` to `ExpertiseApi.csproj` before the first Option B release. This is the load-bearing detail for the portable-single-tarball decision.
- **Roll-forward policy.** Add `<RollForward>LatestMinor</RollForward>` to the csproj so a 10.0.0 tarball survives a host that has only 10.1.x installed. Default `Minor` works but explicit is auditable.
- **`tar` hardening.** Prefer `bsdtar` (libarchive) for extraction — rejects `..` components and refuses symlink-follow by default. GNU tar fallback requires `--no-same-owner --no-same-permissions --no-overwrite-dir --no-xattrs --no-acls --no-selinux` plus a post-extract `find`-based inspector for symlinks/hardlinks/setuid/setgid/device nodes before the OLD↔STAGE swap. Build the tarball deterministically in CI (sorted entries, fixed mtime, owner 0:0, mode bits clamped) so operator-side validation is a confirmation, not a defense.
- **cosign installer pinning.** Mirror `release.yml`'s `sigstore/cosign-installer@<sha>` pin discipline: store `COSIGN_VERSION` + per-RID SHA-256 as constants near the top of `install.sh`. Verify after download with `sha256sum -c`. No `latest` fetches; trust-on-first-use rejected.
- **OIDC identity constants.** `--certificate-identity 'https://github.com/psmfd/agent-expertise-api/.github/workflows/release.yml@refs/heads/main' --certificate-oidc-issuer 'https://token.actions.githubusercontent.com'` — exact match, never `--certificate-identity-regexp`. Same rule the README already documents at the openapi.json verification recipe.
- **Manifest source-of-truth.** `requiredRuntime.minVersion` must be `jq -r .runtimeOptions.framework.version dist/publish/ExpertiseApi.runtimeconfig.json` in CI, not hand-set or copied from `global.json`. `global.json` controls the SDK selected to build/publish; it is not embedded in the apphost or runtimeconfig and does not invalidate already-deployed tarballs when bumped.
- **Reproducibility** is a nice-to-have separate from cosign trust. Adding `Deterministic=true`, `ContinuousIntegrationBuild=true`, `tar --sort=name --owner=0 --group=0 --numeric-owner --mtime=...`, and `gzip -n` lets a third party re-derive the SHA from source. Not required for the signature to bind bytes; pursue separately if SLSA Level 3 becomes scope.
- **Implementation PR** is filed as a follow-up issue alongside this ADR; install.sh code review should additionally dispatch `checkmarx-expert` for SAST over the tar-extraction and cosign-invocation shell code (per security-review-expert's cross-domain note).

## Amendment 1 — blob signatures move to the Sigstore bundle format; pin `cosign-release` (2026-07-10)

**Trigger.** The `v1.4.0` release published partially: the `cosign sign-blob` steps in `release.yml` failed and the portable tarball step was skipped, so the release shipped without the A2 tarball, manifest, and signatures (images + Helm chart were fine). Root cause: the Dependabot bump `sigstore/cosign-installer 3.8.1 → 4.1.2` (#203) changed the installer Action's **default `cosign-release`** from cosign `v2.4.3` → `v3.0.6` (this workflow never pinned `cosign-release`). cosign **v3.0.0** flipped `--new-bundle-format` to default `true`, and **v3.0.6** specifically has a missing-guard bug that turns "new-bundle-format on + no `--bundle`" into `create bundle file: open : no such file or directory`. The workflow was still calling the now-deprecated `--output-signature` / `--output-certificate` flags. (Note: there is no cosign v4 CLI — latest is v3.1.1; the "v4" is only the installer Action.) This is the same class of cosign CLI-surface deprecation recorded in **ADR-012 Amendment 1**, which moved the *backup* path off cosign onto `ssh-keygen -Y`; the release pipeline is the surface that stayed on cosign.

**Decision.** Two coupled changes, tracked in #399:

1. **Artifact format.** `cosign sign-blob` now emits a single self-contained **Sigstore bundle** (`cosign sign-blob --bundle <artifact>.sigstore.json`) that embeds the signature, the Fulcio certificate chain, and the Rekor inclusion proof — replacing the detached `.sig` + `.pem` pair. This supersedes the specific `.manifest.json.sig` / `.manifest.json.pem` and `openapi.json.sig` / `.pem` asset names described in the "Independent GitHub Releases API cross-check" sub-decision above and in the README verification recipes. The GitHub Releases API cross-check now expects **four** assets per tarball release (`*-portable.tar.gz`, `*.tar.gz.sha256`, `*.manifest.json`, `*.manifest.json.sigstore.json`) instead of five. Verify side (`scripts/verify-release.sh`, `scripts/lib/release-consumer.sh`, README) moves to `cosign verify-blob --bundle`, and `COSIGN_MIN_VERSION` rises `2.2.0 → 3.0.0` (the floor that understands the bundle format).

2. **Version pinning (the durable root-cause fix).** `release.yml` and `install-smoke-from-release.yml` now pin `with: cosign-release: v3.1.1` on `sigstore/cosign-installer`, so the cosign CLI version is a reviewed change rather than a side effect of an installer-Action bump. This closes the drift vector that caused the incident. The host-side install path's own `COSIGN_VERSION` pin discipline (implementation note above) is unchanged; a follow-up may extend Dependabot/bootstrap-script coverage to the CLI version (see #399).

**Recovery.** `v1.4.0` remains as published (images and Helm chart are usable; `install.sh --from-release` requires `v1.4.1+`). The fix ships in `v1.4.1`, which is the first release to emit the bundle-format artifact set.
