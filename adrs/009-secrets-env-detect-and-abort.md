# Detect-and-abort over auto-rewrite for legacy unquoted `secrets.env`

- Status: accepted
- Date: 2026-05-19
- Closes: [#160](https://github.com/psmfd/agent-expertise-api/issues/160)
- Companion: [`scripts/install.sh`](../scripts/install.sh) (`maybe_migrate` detector)

## Context and Problem Statement

The bash launch wrapper generated by `scripts/install.sh` sources `secrets.env` via `set -a; . secrets.env; set +a`. This evaluates every line as bash syntax, so any unquoted value containing `;` (the Npgsql connection-string separator) is truncated at the first `;` — the first segment becomes the variable's value, and each subsequent segment is treated as a separate command and exported as its own variable. `systemd`'s native `EnvironmentFile=` parser is literal and tolerates the unquoted form, so the systemd-on-Linux service-start path masks the bug; the launchd-on-macOS path and the `scripts/migrate.sh` invocation both go through the bash sourcer and silently break.

PR #159 hardened three points: the generated stub example shows the **quoted** form (`"Host=…;Port=…;…"`), the install-time `maybe_migrate` step detects the legacy unquoted form and aborts with a remediation-specific error, and `scripts/migrate.sh`/`migrate.ps1` rely on the operator having quoted the value. What PR #159 deliberately did **not** do: silently rewrite an existing operator-authored `secrets.env` to add quotes.

[Issue #160](https://github.com/psmfd/agent-expertise-api/issues/160) asks for either (a) an auto-conversion step in `install.sh` / `install.ps1` that rewrites the legacy line in place during upgrade, or (b) an RCA defending the current detect-and-abort posture as sufficient beyond the A2 archetype's solo-dev scope.

## Considered Options

- **Auto-rewrite** the legacy unquoted line during install. Detect → backup to `.bak` → write the quoted form → continue install. Zero operator action on upgrade.
- **Detect-and-abort** with a remediation-specific error message (current state). Operator edits the file manually and re-runs install.
- **Replace the bash sourcer** with a small parser (Python, awk, or a Go binary) that emits `KEY=VAL` literally without shell evaluation. Defense in depth.
- **Switch to a JSON or YAML `secrets` file** loaded by .NET's configuration pipeline (`AddJsonFile("secrets.json", reloadOnChange: false)`). Well-defined parsing, but breaks the systemd `EnvironmentFile=` chain.

## Decision Outcome

Chosen option: **detect-and-abort** (option 2 — the current state, codified now as a decision rather than left implicit).

### Why not auto-rewrite

- **Silent mutation of an operator-authored secrets file is the wrong default for any file under `/etc`-equivalent semantics.** The operator's mental model of `secrets.env` is that its contents are stable across `install.sh` invocations unless they explicitly edited it. A rewrite — even with a `.bak` companion — violates that contract and surfaces as a surprise on the next `diff` or VCS check.
- **Backup hygiene is non-trivial.** A `.bak` companion in a `chmod 600` file is itself a `chmod 600` artifact containing the same secret material. Operators routinely back up directories without distinguishing `.bak`-from-config and `.bak`-from-runtime, leading to long-lived copies of credentials in tarballs and snapshots. Detect-and-abort produces zero new file artifacts.
- **The remediation cost is bounded and one-time.** The detect-and-abort message names the exact line shape and provides the corrected form. An operator who has reached the install-time error has the file open; the fix is a single edit followed by re-running `install.sh`. The detector covers the upgrade flow exactly once per host per fix.
- **The bug class is closed at source for new installs.** `ensure_config_stubs` writes the quoted-form stub from PR #159 onward, so new installs never carry the legacy shape. The detect-and-abort path serves only operators upgrading from pre-#159 installs.

### Why not replace the bash sourcer

- The wrapper script is generated by `install.sh` and is a tight, auditable artifact (≈ 20 lines). Introducing a parser binary (Python, awk, Go) widens the install-time dependency surface and contradicts the project's portable-by-default posture. Python is not guaranteed on minimal Debian/macOS installs; awk is, but a hand-rolled awk parser for KEY=VAL with quoting semantics re-implements `Microsoft.Extensions.Configuration.EnvironmentVariables` behaviour poorly.
- The systemd path already uses `EnvironmentFile=`, which has correct parsing. Defense in depth here would add code on the path that is *already* tolerant of the bug, not on the path that surfaces it.

### Why not move to a JSON `secrets` file

- Larger lift. Touches the .NET configuration loader, the systemd unit template (loses `EnvironmentFile=`), the install/upgrade docs, the `migrate.sh` script, and the README.
- Breaks the systemd `EnvironmentFile=` contract that operators currently rely on for orchestration (e.g., `systemctl edit --full` to override a single variable for a one-off debug session).
- Trades one parsing footgun for a different one (JSON parsing errors are equally hostile to operators) without addressing the actual root cause (operators-copy-pasting-unquoted-stubs).

## Scope-of-applicability note (per the issue)

The A2 archetype (solo-dev, single API process, no PgBouncer) is the smallest deployment shape this project supports. The detect-and-abort posture is defensible beyond it for one reason that holds across all archetypes:

- The **only** legitimate source of the legacy unquoted shape is a pre-#159 install. Greenfield installs (A2, A3, k8s) start from the quoted-form stub. Upgrades from pre-#159 are a finite, one-time event per host. Auto-rewrite optimises for a one-time path at the cost of perpetually accepting silent-mutation semantics on a credentials file — a poor trade.

**A3 (multi-process / PgBouncer, native install)** inherits A2's `secrets.env` shape verbatim — PgBouncer is a separate process with its own `userlist.txt`, and the .NET API process still reads `ConnectionStrings__DefaultConnection` from `secrets.env` via the same wrapper. The same argument applies without modification.

For deployments where operator self-service edits to `secrets.env` are not the working pattern (e.g., k8s with `secretGenerator` / external-secrets / sealed-secrets feeding a `Secret` mounted as `secrets.env`), the rendered file is produced by the secret-management tool and the detect-and-abort path is unreachable: the secret manager produces the quoted form because the stub it was modelled on is the quoted form.

### The detector error an operator sees

When `scripts/install.sh maybe_migrate` encounters the legacy shape, it aborts with this literal message (text reproduced here so this ADR is grep-discoverable from the error itself):

```text
detected legacy unquoted ConnectionStrings__DefaultConnection in <SECRETS_FILE>:
the bash sourcer split on ';' and only kept the Host= segment.
Wrap the value in double quotes (e.g.,
ConnectionStrings__DefaultConnection="Host=...;Port=...;Database=...;Username=...;Password=...")
and re-run scripts/install.sh.
```

The remediation is one edit (add surrounding double quotes) followed by re-running `scripts/install.sh`. No other state needs to be reset.

## Consequences

### Positive

- `secrets.env` is never mutated by `install.sh`. Operator's mental model of `/etc`-equivalent semantics is preserved.
- No `.bak` files containing credential material are written to disk.
- Detector is local to `maybe_migrate` (≈ 6 lines of bash) and continues to work for any future variant of the same footgun without re-architecting the configuration loader.

### Negative

- Operators upgrading from a pre-#159 install with a legacy unquoted line see one extra install failure with a remediation message. They edit the file and re-run. One-time cost per host.
- The detector is heuristic (`Host=` present, no `;`, separate `Port=` line nearby). False positives are conceivable if an operator intentionally splits a connection across multiple lines, but the documented stub form is single-line-quoted and we have not observed multi-line variants in practice.

### Revisit triggers

Re-open this ADR if any of the following hold:

- The wrapper script grows to need substantive parsing logic for other reasons (multi-line values, conditional defaults, secret rotation), making a non-bash parser cost-justified.
- A first-class secret-management integration (Vault, AWS Secrets Manager, Azure Key Vault) becomes the recommended pattern for the A3+ archetypes, in which case the wrapper sources a different file shape entirely.
- ≥ 3 distinct operators file issues reporting the detect-and-abort UX as an upgrade blocker (not merely as feedback) within a single release cycle, OR a single operator demonstrates a deployment topology where re-running `install.sh` is non-trivial (e.g., orchestrator-managed install where the file edit must round-trip through a separate config-management system).
